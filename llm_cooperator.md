# 🏗️ 多模型协作问答优化系统架构设计

**架构愿景**：通过多模型协作（Multi-Agent Swarm）与大模型裁判（LLM-as-a-Judge）机制，构建一个具备自我纠错、多维验证和动态仲裁能力的高可靠问答系统。
**核心理念**：两种协作模式 + 三层架构 + 仲裁决策

---

## 模式一：主从模式（Primary-Secondary Mode）

> **定位**：推荐的默认模式。侧重于高效、精准的迭代优化，适用于对准确性要求高的常规任务（如代码生成、事实问答、逻辑推演）。

### 1. 运行流程

`用户输入` ➔ `主模型生成` ➔ `副模型并行检测` ➔ `仲裁决策与修订` ➔ `输出最佳答案`

### 2. 三层架构详解

* **生成层（主模型）**
* 负责接收用户输入（包含上下文、历史对话等），生成高质量的**初稿答案**，并附带**置信度估计**（如 置信度：0.72）。


* **验证层（副模型集群 - 并行处理）**
* 采用异构分工机制，多个副模型戴上不同的“面具（System Prompt）”并行审查初稿。每个副模型需输出：结论、问题点、证据/理由、置信度、修改建议。
* **事实核查模型**：负责事实正确性、知识检索验证、数据/引用检查。
* **逻辑推理模型**：负责推理解致性、逻辑漏洞检查、步骤完整性。
* **安全与风险模型**：负责安全性检查、偏见/风险识别、合规性检查。
* **风格与意图模型**：负责风格匹配、意图对齐、表达优化。


* **仲裁层（聚合决策）**
* **聚合器（Judge 模型 / 规则引擎）**：收集所有副模型的意见，计算综合评分，判断是否需要修改，并制定具体的修改策略（如：主要问题是缺少数据来源，策略为补充引用并优化表达）。



### 3. 最终输出

* 输出经过优化后的**最佳答案**。
* （可选）提供修改说明：如补充了哪些数据来源、优化了哪些表达方式。

---

## 模式二：圆桌模式（Roundtable Mode）

> **定位**：增强模式。侧重于集思广益和观点碰撞，适用于复杂、发散性、争议性大或开放性的任务（如架构设计、头脑风暴、哲学思辨）。

### 1. 运行流程

`多模型独立生成` ➔ `观点交互（可选）` ➔ `评估与投票` ➔ `仲裁决策` ➔ `输出最佳答案`

### 2. 核心机制详解

* **多模型独立生成（第一轮）**：针对同一问题，模型 A、B、C...N 分别独立生成自己的回答。
* **观点交互（可选环节）**：各模型之间相互质疑、补充、辩论（可配置辩论轮数）。
* **评估与投票**：
* 对每个模型的回答进行**多维度打分**（准确性、逻辑性、完整性、相关性、清晰性、安全性）。
* 通过**加权投票 / 加权求和**得出各模型的综合得分。


* **仲裁决策**：
* 由综合评分最高者胜出，或者交由高阶的 Judge 模型进行终局裁决。
* 支持融合部分次优模型的优秀观点。


* **回退机制（Fallback）**：若全员置信度不足或分歧过大，可触发退回第一轮重新生成，或转入“主从模式”进行精修。

---

## ⚙️ 系统核心底层逻辑 (Core Mechanisms)

### 1. 关键评估信号（仲裁依据）

系统在做决策时，不依赖单一指标，而是综合考量以下信号：

* **一致性**：副模型之间对主要观点是否达成一致。
* **证据支持度**：是否具备外部检索结果或上下文的强力支撑。
* **冲突程度**：指出的错误有多严重（致命错误 vs 格式瑕疵）。
* **置信度**：各模型对自身结论的把握程度。
* **任务类型**：针对事实、推理、写作等不同任务动态调整关注点。
* **成本与时延**：兼顾模型调用的 Token 成本和响应时间。

### 2. 仲裁策略与决策规则 (示例)

* **打分融合公式**：$Score = \sum (w_i \cdot S_i)$
*(其中 $S_i$ 为第 $i$ 个维度的评分(0-1)，$w_i$ 为该维度的权重，可根据任务动态调整)*
* **决策动作路由规则**：
1. 若 $Score \ge T_{high}$ ➔ **保持原答案**（主模型答案已足够好）。
2. 若 $T_{low} \le Score < T_{high}$ ➔ **局部修改 / 补充信息**（针对特定问题精确修正，或补充缺失的来源/数据）。
3. 若 $Score < T_{low}$ ➔ **重写**（生成全新的答案）。
4. 若各模型分歧过大（方差 $>$ 阈值）➔ 自动升级进入**圆桌模式**进行深度探讨，或输出**无法确定**的说明。

## 🌟 系统优势与落地建议

### 系统优势

* **提高准确性与可靠性**：通过交叉验证大幅降低单一模型的“幻觉（Hallucination）”和逻辑错误。
* **可解释的决策过程**：每一步的质疑、打分、修改建议都有迹可循。
* **可扩展、可配置**：副模型的面具和权重可以根据业务需求任意插拔和增减。
* **平衡效果、成本与时延**：通过阈值判断避免无效的重复生成，将昂贵的计算资源用在刀刃上。

### 💡 架构师调度建议 (Best Practice)

在实际工程落地中，默认使用【主从模式】**以平衡生成效果与调用成本；只有当系统探测到问题极度复杂、存在巨大争议，或者主从模式的最终置信度依然不足时，再**动态切换到【圆桌模式】以获取更全面、深度的答案。

太棒了，进入到这一步，我们就正式踏入了高级 AI 工程（AI Engineering）的深水区。

要实现我们刚刚设计的“主从模式（Maker-Checker Pipeline）”，市面上有像 LangGraph、AutoGen 这样的重型框架。但是，考虑到我们已经构建了一套极其精巧、轻量且高度定制化的 FastAPI + SSE 流式系统，**我不建议引入这些重型框架**（它们会破坏我们现有的控制流和 BYOK 机制）。

作为 Tech Lead，我为你设计的方案是：**采用“原生异步编排” + “结构化输出（Structured Outputs）”的轻量级状态机算法。**

以下是我们将要采用的核心技术与算法选型：

### 🛠️ 核心工具与算法选型

#### 1. 并发调度算法：`asyncio.gather` (原生 Python)

* **用途**：实现副模型的**并行**审查。
* **逻辑**：当主模型生成初稿后，我们不需要等“逻辑审查员”看完再给“安全审查员”看。我们将使用 `asyncio.gather(critic_a(), critic_b(), critic_c())` 同时唤醒多个副模型，极大缩短审查阶段的耗时（时延由最慢的那个副模型决定，而不是总和）。

#### 2. 强制结构化输出 (JSON Schema / Pydantic)

* **用途**：解决“大模型回答太发散，无法用代码做算术题”的痛点。
* **算法**：我们将利用 OpenAI 协议的 `response_format={ "type": "json_object" }` 强制副模型（Critic）必须返回严格的 JSON 数据。
例如，强制它们输出如下格式：
```json
{
  "pass": false,
  "score": 65,
  "criticism": "缺少数据来源支持",
  "suggestions": ["补充2023年的统计数据", "修改过于绝对的语气"]
}

```

只有拿到这样结构化的数据，我们的后端 Python 代码才能算出平均分，并用 `If/Else` 做出下一层的路由决策。

#### 3. 状态机循环算法 (Finite State Machine with Fallback)

* **用途**：控制主从迭代的流转，防止死循环。
* **算法**：
* 设定 `MAX_ITERATIONS = 3`。
* 如果聚合评分 `Average_Score >= 85` 或 `All_Pass == True`，跳出循环。
* 否则，将所有 `suggestions` 打包，拼接到系统提示词中，触发下一次循环 `iteration += 1`。
* 如果达到最大迭代次数依然不及格，则强制输出当前最优版本，并附带一条“置信度警告”。



#### 4. SSE 状态流式代理 (Stateful SSE Proxy)

* **用途**：控制前端 UI 的显示。在内部循环打架时，前端不应看到乱码。
* **算法**：我们在后端的 SSE 生成器中，新增一种事件类型 `{"type": "status", "message": "🕵️ 正在进行并发审查 (逻辑、安全)..."}`。前端接收到这个事件后，不打印在聊天框里，而是用一个 UI 提示条（Toast 或 Spinner）展示当前系统的心智状态。只有达到终局时，才抛出 `{"type": "chunk", ...}`。
